Executive Summary

GetDraft's frontend and backend are both built: all eight client change requests (CRs) are checked off in the sprint plan, the React Native app ships every screen with a graceful mock-data fallback, and the NestJS backend has all 12 modules, 8 SQL migrations, and a test suite in place. The remaining work is not "build features" — it is verify, connect, harden, and ship.

This plan is grounded in a direct scan of the repository rather than the sprint checkboxes, which surfaced three things worth acting on immediately:

  • The backend is closer to live than the docs say. The .env already holds real Supabase, Stripe, SMTP, Twilio (phone verification) and Didit (identity verification) credentials. Only SUPABASE_JWT_SECRET and EXPO_ACCESS_TOKEN are empty.
  • Scope has grown beyond the original Notion plan — phone verification (Twilio) and KYC/identity verification (Didit) now exist and a migration 008 was added. These need their own verification and aren't reflected in the sprint doc.
  • The genuine gaps are integration and QA: nobody has run the app against the live backend end-to-end, and device testing was done on the iOS simulator only — physical iOS and Android testing is outstanding.

2 · Gap Analysis — What Actually Remains

Five buckets of work stand between the current state and a client-demoable, launch-track build.

G1 — Frontend verification & hardening

Trust the code, not the checkboxes. Audit each of the 8 CRs against its acceptance criteria, confirm the stats screen exists and is wired into the welcome flow, and confirm the globe holds its 60fps / <100MB budget on real hardware (covered in QA).

G2 — Backend go-live

Apply the 8 migrations to the Supabase project, fill the two empty secrets, verify all 12 modules boot, run the e2e suite against live creds (it currently auto-skips), and deploy to Railway with a stable public URL.

G3 — Live integration (remove the mock crutch)

Point the app at the deployed API, walk every role's flow against real data, and confirm the graceful fallbacks stay as a safety net but are no longer doing the heavy lifting. Validate the business rules (parent → 403 on discover, 11th basic swipe → 429, auto-downgrade on cancel).

G4 — Newly-discovered scope: Twilio + Didit

Phone verification and identity/KYC verification exist in the backend but aren't in the sprint plan. They need their own test pass, frontend wiring confirmation, and a documented decision on whether they're in-scope for the client demo.

G5 — Cross-platform QA & launch prep

Physical iOS + Android testing, performance budgets, EAS demo build, and store-readiness items (icons, privacy policy, listings) — plus reconciling the project docs with reality.

6 · Definition of Done

The project is "demo-ready" when all of the following hold simultaneously:

  • All 8 CRs verified against acceptance criteria in code, not just checked off — with screenshots.
  • Backend deployed to a stable public URL; all 12 modules boot; e2e suite passes against live Supabase.
  • App runs end-to-end on the live API for all four roles with the mock fallback disabled.
  • The three business rules (parent 403, basic-swipe 429, auto-downgrade) observed live.
  • Stripe test-mode checkout + webhook + downgrade verified.
  • Globe meets 60fps / <100MB / <8s cold-start on a physical iPhone 12+ and Pixel 6+.
  • Push notifications received on a physical device for both platforms.
  • EAS demo builds installable on iOS + Android; scripted demo runs without fallback.
  • Secrets out of the repo and into Railway vars; .env git-ignored; keys rotated if exposed.
  • Docs (TODO/sprint/handoff) reconciled with the shipped reality, incl. Twilio/Didit.
